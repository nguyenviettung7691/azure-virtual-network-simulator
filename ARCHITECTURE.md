# Architecture & Component Specification: Azure Virtual Network Simulator

This document serves as the exhaustive implementation specification and developer reference for the feature set of the Azure Virtual Network Simulator. It outlines the architectural decisions, component behaviors, UI/UX implementation details, and state-management rules required to build and maintain the application.

---

## 1. Core Canvas Implementation

### 1.1 Diagram Engine (Vue Flow)
- **Framework:** `@vue-flow/core` serves as the primary diagram engine.
- **Node Constraints:** 
  - Supports 27 specific Azure networking components (user-managed).
  - Canvas must disable manual edge creation, reconnection, and deletion. Edge interactivity is strictly read-only.
- **System-Managed Entities:**
  - `Public Internet`: Automatically injected into the state when `userNodeCount >= 1`.
  - Must have an immutable state (read-only, cannot be deleted manually, does not trigger a configuration form on click).
- **Edge Rendering:**
  - **Attachment Edges:** Automatically generated by mapping component relationships (e.g., NSG to Subnet). Rendered as visible arrows.
  - **Containment Edges:** Derived directly from the `node.parentNode` attribute. They express visual nesting on the canvas rather than rendered arrows.

### 1.2 Layout & Viewport Controls
- **Fit Content:** Pans and zooms the canvas viewport to calculate and encompass the bounding boxes of all rendered nodes.
- **Fit View:** Resets the viewport transform to default values (`x: 0, y: 0, zoom: 1`).
- **MiniMap Interaction:** Minimap supports direct interaction (`pannable`, `zoomable`) so users can drag in the minimap to pan and use wheel-scroll on minimap to zoom the main viewport.
- **State Toggles (Lock/Unlock):**
  - **Locked (Default read-only):** Pan and zoom enabled. Node dragging, selection, and property editing disabled. Bound to a visually highlighted toggle button.
  - **Unlocked (Edit mode):** Enables node dragging and selection. `cursor: pointer` applied on component name hover. Triggers the configuration form sequence on `click`.

## 2. Component Configuration & Property Forms

- **Trigger:** `onClick` event on any user-managed node (requires canvas state `Unlocked`).
- **Target:** Opens the corresponding property edit form within the Right Side Panel.
- **Validation:** Forms must validate Azure-specific rules (e.g., CIDR overlaps, naming conventions). Form states are dynamically mapped to the specific `NetworkComponentType` clicked.

## 3. UI Panels & Data Visualization

### 3.1 Network Summary Panel (Right Side Panel)
A reactive dashboard reflecting the live diagram state.

- **Accordion default expansion:** Only the `Components` accordion panel is expanded by default; `Connectivity`, `Security`, and `Performance` start collapsed.

#### 3.1.1 Components Section
- **Grouping:** Nodes grouped by component type.
- **Headers:** Clickable `.group-header-clickable` per type — includes component icon, label, count `<Tag>`, and a chevron icon. Clicking toggles `collapsedComponentTypes` (a `Set<string>` ref) to collapse/expand the node list below via `v-show`.
- **Default state:** All component-type groups initialize collapsed by default.

#### 3.1.2 Connectivity Section
- **Structure:** Div-based collapsible groups (`.conn-groups` container), one `.conn-group` per target node.
- **Data Synthesis:** Rows reflect explicit attachment edges *and* synthesized containment relationships mapped from `node.parentNode`, guaranteeing full topology visibility.
- **Collapse/Expand:** Each `.conn-group-header` (chevron + target icon + name + source count badge) is clickable — toggles `collapsedConnTargets` ref; the `.conn-sources` list of source rows is shown/hidden via `v-show`.
- **Iconography:** Source and Target cells include a small, colored component-type icon. Tooltips expose the full type label.
- **Default state:** All target groups initialize collapsed by default.
- **Count badge contrast:** The source-count badge uses a solid primary background with primary-contrast text for readability.

### 3.1.3 Security and Performance Sections
- **Count badges:** Each component-type count `<Tag>` uses dynamic `:severity` via `getCountSeverity(types, findings)`:
  - `'secondary'` — no nodes of this type in the diagram
  - `'danger'` — at least one `critical` finding with a matching `relatedTypes` entry
  - `'warn'` — at least one `warning` finding matches (and no critical)
  - `'success'` — nodes exist and no relevant findings
- **`AuditFinding`** type includes `relatedTypes?: NetworkComponentType[]`; every `findings.push()` call in `securityFindings` and `performanceFindings` sets this field so `getCountSeverity` can filter correctly.

## 4. Testing & Validation Engine

### 4.1 Test Types
- **Connection (Internet -> VM):** Validates NSG rules and routing.
- **Load Balance:** Validates backend pool paths (Source -> LB/App Gateway -> Backend VMs).
- **DNS Resolution:** Validates DNS Client -> Zone Name -> Resolved IP.
- **Security Audit:** Validates NSG coverage and policy configuration (Outputs list of findings vs visual path).

### 4.2 Network Tests Panel UI
- **Action Buttons:** 
  - Row 1: Full-width "Add Test" block button.
  - Row 2: Half-width "Run All" (`severity="success"`) and "Delete All" (triggers confirm dialog).
- **Summary Tags:**
  - Row 1 (centered): Pass, Fail, Warning.
  - Row 2 (centered): Total.
- **Test Cards:**
  - Visuals: Left-accent border.
  - Header Row: Type icon + Test Name + Bottom border separator.
  - Status Row: Left-aligned `<Tag>` for status (Pass/Fail/Pending, uppercase, center-aligned text within tag). Right-aligned action buttons (Run, Run Animation, Edit, Delete).
  - *Note:* "Run Animation" button is rendered for **connection, load-balance, and DNS** test types.
- **Results Output:**
  - Order: Summary Message -> Detailed Findings (if applicable) -> Compact Inline Flowchart.
  - **Flowcharts:** Renders literal physical traversal paths (Node ID resolving to Component Name + Icon).
    - *Passing:* Final node highlighted green with `mdi:check-circle` badge (`.fni-success` / `.flow-success-badge`).
    - *Failing:* Path terminates explicitly at the blocking component (e.g., NSG) with a red `mdi:close-circle` badge (`.fni-blocked` / `.flow-blocked-badge`).
    - *Warning:* Final node highlighted yellow with `mdi:alert-circle` badge (`.fni-warning` / `.flow-warning-badge`, using `var(--yellow-500)`). Applies to all test types (connection, loadbalance, dns).

### 4.3 Connection Flow Animation Mode
- **State Switch:** Canvas toggles from **Infrastructure** view to **Animation** view.
- **Edge Style:** All animation-mode edges use `getSmoothStepPath` (right-angle paths with `borderRadius: 12`) from `@vue-flow/core`. Active-state edges display marching animated dashes (`stroke-dashoffset` animation via `@keyframes animationEdgePulse` defined globally in `assets/css/diagram.css`). Pending-state edges display static dashes. Terminal-state edges (pass/fail/warning) display solid colored strokes.
- **Execution by test type:**
  - **Connection:** Traveler follows the hop-by-hop physical path from source to target sequentially.
  - **Load-balance:** Phase 1 — one traveler travels source → LB. Phase 2 — multiple travelers depart the LB node simultaneously, one per backend VM, all in parallel (`playParallelSegments` store action creates all overlay edges with `travelerVisible: true` at once, waits one `segmentDurationMs`, then applies terminal state). Both phases use the same default per-hop pacing as other test types (`DEFAULT_ANIMATION_SEGMENT_MS`, currently 1600 ms) unless explicitly overridden.
  - **DNS:** Raw resolved-IP path entries are resolved to real node IDs before animation; unresolvable entries are dropped.
- **Exit:** A fixed, top-centered `Exit animation` button restores the `Infrastructure` view.

## 5. Auto-Layout Engine (Dagre)

- **Execution:** On-demand layout processing triggered manually via the UI (not reactive to every node addition).
- **Algorithm sequence per run:**
  - **Stage 1 - Mathematical prerequisites (Kahn topological sort, BFS):**
    - In-degree and adjacency maps are built from layout-relevant directed edges.
    - Nodes with zero in-degree are dequeued repeatedly to produce deterministic dependency order.
    - Parent-before-child precedence is enforced for Vue Flow nesting correctness.
    - If cycles exist, unresolved nodes are appended in deterministic ID order so layout remains total-order and non-blocking.
  - **Stage 2 - Hierarchical placement (Sugiyama model via Dagre):**
    - Rank direction is left-to-right (`rankdir: LR`) to emphasize traffic/data flow.
    - Dagre computes initial absolute positions before semantic reflow passes.
    - Root-VNet spacing normalization enforces deterministic minimum gaps and exact row-to-row Y rhythm (both pull-up and push-down correction) so repeated manual Auto-Layout runs do not collapse VNets onto the same coordinates.
  - **Stage 3 - Orthogonal routing policy:**
    - Rendering uses right-angle smoothstep edges.
    - **Layer classification:** Components are semantically classified into 4 vertical bands: `System-Managed` (Public Internet node) → `Public-Facing` (internet edge services) → `VNet` (network fabric) → `Private` (backend PaaS).
    - **Classification rules:**
      - **Always Public-Facing:** VPN Gateway, Bastion, Public IP Address, Public DNS Zone, Public Load Balancer, Public App Gateway, public App Service, public Azure Functions
      - **Always VNet:** VNet, Subnet, VNet Peering, NIC, NSG, ASG, Firewall, UDR, NVA, VM, VMSS, AKS, Internal Load Balancer, Internal App Gateway, Service Endpoint, Private Endpoint
      - **Always Private:** Storage Account, Blob Storage, Managed Disk, Key Vault, Managed Identity, Private DNS Zone
      - **Config-driven:** App Gateway (frontendType: 'Public' | 'Internal'), Load Balancer (loadBalancerType: 'Public' | 'Internal'), App Service/Functions (VNet Integration or Private Endpoint presence), AKS (node pools in VNet, API server can be public/private)
    - **Edge routing by layer relationship:**
      - Higher → lower layer: source exits `Bottom`, target enters `Top` (e.g., Public Load Balancer → Private backend NIC)
      - Same layer or ambiguous layer classification: source exits `Right`, target enters `Left` (e.g., NSG → Subnet within VNet layer)
      - Lower → higher layer: source exits `Top`, target enters `Bottom` (e.g., Private Storage → VNet Service Endpoint)
    - Routing decision is semantic-first by layer classification. Same-layer and ambiguous cases intentionally remain horizontal (`Right` → `Left`).
- **Edge style:** Both `NetworkEdge.vue` and `AnimationEdge.vue` use `getSmoothStepPath` (right-angle orthogonal paths, `borderRadius: 12`). This reduces visual overlap with container nodes compared to bezier curves.
- **Architecture-mode edge contrast:** Infrastructure edges (`NetworkEdge.vue`) use a theme token (`--diagram-edge-color`) for high contrast against the canvas: dark neutral in light mode and near-white in dark mode. Edge labels and arrow markers follow the same token so they remain visually consistent.
- **Uniform node spacing:** Auto-layout uses a shared 32 px inter-node gap for Dagre rank/node separation and attachment reflow spacing to keep edge arrowheads readable between adjacent nodes.
- **Positioning Rules:**
  - `Public Internet`: Isolated, top-row dedicated placement above all topology.
  - `VNet`: Expands to visually enclose all child Subnets.
  - `Subnets`: Stays strictly bounded inside parent `VNet`. Dynamically resizes to pack contained workloads.
  - `Workloads`: Kept strictly within parent `Subnet` bounds.
  - `NICs`: Associated workloads are laid out horizontally *beside* the NIC (preventing vertical stacking underneath).
  - `Policy Resources (NSG, ASG, UDR)`: Positioned outside the `VNet` but physically aligned to their protected Subnet/NIC.
  - `Public-facing root nodes`: Placed in compact rows in a dedicated lane above NSG/ASG/UDR by `reflowPublicFacingNodes` (Public IP, VPN Gateway, Bastion, public DNS, public LB/App Gateway, and public App Service/Functions).
  - `Root VNet top-band clearance`: `enforceRootVnetTopBandClearance` runs after public-facing reflow and pushes any root VNet that drifts upward back below top lanes, including its descendants, preventing overlap with public/policy layers.
  - `Root VNet spacing`: `compactRootVnetSpacing` enforces deterministic minimum horizontal and vertical gaps between root VNets on every run, and normalizes row-to-row Y spacing in both directions, to prevent rerun overlap.
  - `Root VNet-managed nodes`: `reflowRootVnetManagedNodes` places root VNet-layer non-container nodes in compact rows below VNet content.
  - `VNet Peering`: Uses Local-VNet containment (`parentNode = localVnetId`). `reflowVnetPeeringNodes` only repositions unparented peering nodes between VNets as a fallback behavior.
  - `Root private infrastructure nodes`: Placed in compact rows immediately below the main content bounding box (40 px gap, 32 px between nodes) by `reflowRootInfrastructureNodes`.
  - *Density Handling:* Outside-VNet policy lanes are staggered automatically if anchor points are dense to minimize overlapping edge lines.
- **Padding:** Top clearance is calculated and reserved to prevent nodes from clipping VNet and Subnet text headers.

## 6. Seed Data & System Defaults

### 6.1 Sample Architectures (`Quick Sample` and `Full Sample`)
- **Empty-state buttons:** `Add VNet`, `Quick Sample`, `Full Sample`.
- **Quick Sample action:** Seeds the baseline architecture.
- **Full Sample action:** Builds on top of Quick Sample, then appends all remaining supported Azure component types plus additional tests.

#### 6.1.1 Quick Sample
- **Topology size:** 17 nodes.
- **Components:**
  - 1x VNet (`region: southeastasia`)
  - 3x Subnets
  - 3x NSGs (explicit allow/deny port-80 rules plus deny-all)
  - 3x NICs (static private IPs)
  - 4x VMs
  - 1x Standard Public IP
  - 1x Standard Public Load Balancer
  - 1x Private DNS Zone linked to the VNet
- **Seeded Tests:** 6 tests total (4x connection, 1x load-balance, 1x DNS).
- **Layout contract:** Uses attachment edges only (no containment edges), then executes `autoLayout()` and `notifyDiagramLoaded()`.

#### 6.1.2 Full Sample
- **Base strategy:** Calls the Quick Sample loader first, then appends the rest of the component catalog in one deterministic pass before the queued layout runs.
- **Additional component coverage:** ASG, UDR, VPN Gateway, App Gateway, NVA, VMSS, AKS, App Service, Functions, Storage Account, Blob Storage, Managed Disk, Managed Identity, Key Vault, Service Endpoint, Private Endpoint, Firewall, Bastion, VNet Peering, plus an additional VNet/Subnet for peering context.
- **Additional tests:** Adds targeted tests beyond Quick Sample for:
  - Application Gateway load-balancing path.
  - Workload-to-private-endpoint connectivity.
  - Internet-to-Bastion path.
  - Additional private DNS resolution.

#### 6.1.3 Implementation Plan (Developer Detail)
1. Keep `loadQuickSampleDiagram()` as the canonical baseline seed routine.
2. Implement `loadFullSampleDiagram()` as an extension layer that:
   - reuses baseline IDs where needed,
   - appends missing component types,
   - and creates only attachment edges.
3. Ensure seeded nodes provide required data fields expected by summary audits and test simulation logic (for example NSG rules, LB/AppGW backend references, DNS record sets).
4. Add full-sample-specific tests using existing test types (`connection`, `loadbalance`, `dns`) so they auto-run through the current `diagramLoadId` watcher path.
5. Preserve single-pass post-seed behavior: one Auto-Layout run and one diagram-loaded notification to avoid duplicate test reruns.

### 6.2 Data Destruction
- **Reset Diagram:** Removes all components. A confirm dialog must provide an optional checkbox: "Also reset all network tests".
- **Delete All Tests:** Requires confirmation. Flushes test state without altering canvas components.

- Integration with Bedrock evaluates diagram state in real-time.
- Supports 4 difficulty tiers. Generates time-boxed prompts against which the local canvas state is verified for compliance.

## 8. Styling Architecture

- **Theme Mode:** Dark/Light mode strictly respects system OS preferences using media queries, with a manual override toggle.
- **CSS Hierarchy:**
  - **Application Tokens:** Layout, sizing, and foundational base colors are governed by vanilla CSS variables located in `assets/css/main.css`.
  - **Semantic Tokens:** Component surfaces, text themes, and state colors are governed by PrimeVue Aura semantic tokens.
  - **Redundancy:** `main.css` must mirror PrimeVue semantic tokens as fallback aliases, ensuring layout integrity prior to the runtime injection of the PrimeVue theme layer.

## 9. Deployment & Operations Architecture

### 9.1 Hosting Model
- The application is a static, client-rendered SPA (`ssr: false` in `nuxt.config.ts`).
- Runtime execution happens in the browser only; hosting infrastructure serves static assets.
- Build artifacts:
  - `npm run build` -> `.output/` (Nuxt production build)
  - `npm run generate` -> `.output/public/` (static artifact for CDN hosting)

### 9.2 Canonical Production Topology (Amplify Origin + CloudFront Front Door)
- **Canonical deployment target:** CloudFront in front of AWS Amplify Hosting.
- Reference topology:
  - Browser -> CloudFront distribution -> Amplify Hosting app URL (origin)
  - Browser directly calls external services (Cognito, S3 APIs via Amplify, Bedrock Runtime, MongoDB Data API)
- Rationale:
  - Keeps Amplify native Git-based CI/CD unchanged for application lifecycle.
  - Supports custom-domain front-door control through CloudFront and Route 53.
  - Decouples application releases from edge/network infrastructure lifecycle.

### 9.3 TLS, DNS, and Domain Requirements
- ACM certificate must be issued in `us-east-1` for CloudFront.
- Certificate validation is performed via DNS records in Route 53.
- CloudFront Alternate Domain Name (CNAME) is set to the app custom hostname.
- Route 53 points the app hostname to the CloudFront distribution domain.
- Amplify still requires SPA rewrite behavior (`/<*>` -> `/index.html` as `200`) on the app branch so deep links resolve correctly when proxied through CloudFront.

### 9.4 Event-Driven Cache Invalidation (Required)
- Problem: Amplify deploys and CloudFront cache lifecycle are independent; edge content can remain stale after deploy.
- Mandatory glue:
  - EventBridge rule listens for successful Amplify deployment events.
  - Event pattern:
    - `source = aws.amplify`
    - `detail-type = Amplify Deployment Status Change`
    - `detail.jobStatus = SUCCEED`
  - Target: Lambda function.
  - Lambda action: `cloudfront:CreateInvalidation` with path `/*` for the managed distribution.
- Outcome: every successful Amplify deploy automatically triggers CloudFront cache refresh without GitHub-side credentials.

### 9.5 Infrastructure Ownership and IaC Boundary
- App lifecycle owner: Amplify native CI/CD (Git push drives build/deploy).
- Infrastructure lifecycle owner: Terraform in `infra/`.
- Terraform-managed resources include:
  - CloudFront distribution
  - ACM certificate and validation records
  - Route 53 records for app hostname
  - EventBridge rule for Amplify deployment-success events
  - Lambda invalidation function
  - IAM permissions for Lambda (`cloudfront:CreateInvalidation`)
- Required operator prerequisites before `terraform plan/apply`:
  - Terraform CLI version `>= 1.6.0`
  - AWS CLI v2 is installed and available (`aws --version` succeeds)
  - AWS credentials for the target account are configured and valid (`aws sts get-caller-identity` succeeds)
  - Human access should use short-term credentials (IAM Identity Center preferred): requires IAM Identity Center user, target-account permission set assignment with scope for ACM/Route 53/CloudFront/Lambda/EventBridge/IAM/CloudWatch Logs, and refresh via `aws sso login` when expired
  - Preflight checks from `infra/scripts/check-prereqs.ps1` or `infra/scripts/check-prereqs.sh` are recommended as the standard readiness gate

### 9.6 Cache-Control Strategy
- `index.html` should be short-lived (`no-cache` or very low TTL) to pick up new release manifests quickly.
- Fingerprinted bundles under `_nuxt/*` should be long-lived (`max-age=31536000, immutable`).
- On each successful Amplify deployment, run automatic CloudFront invalidation `/*` via the EventBridge -> Lambda path.

### 9.7 Environment & Configuration Strategy
- Maintain isolated environment stacks: `dev`, `staging`, `production`.
- Each environment should use distinct values for:
  - Cognito user pool/client IDs
  - S3 bucket
  - Bedrock region (if applicable)
  - MongoDB endpoint/API key/database/collection
- Critical rule:
  - `NUXT_PUBLIC_*` values are embedded at build time in the client bundle.
  - Changing env vars in hosting without rebuild/redeploy does not change active runtime behavior.

### 9.8 Release Flow
1. Merge to the release branch (`main` for production).
2. Amplify native CI/CD runs `npm ci` and `npm run generate` and deploys to Amplify Hosting.
3. Amplify emits deployment status event on successful completion.
4. EventBridge matches the success event and invokes Lambda.
5. Lambda creates CloudFront invalidation `/*`.
6. Smoke-test sign-in, save/load setup, challenge generation, settings persistence, and fresh-content delivery.

### 9.9 Rollback Flow
- Application rollback:
  - Redeploy a previous successful Amplify build from deployment history.
- Edge rollback/freshness control:
  - Trigger CloudFront invalidation `/*` after rollback deployment to ensure stale edge cache is replaced.

### 9.10 Security Hardening for Hosted SPA
- Enforce HTTPS-only access.
- Add response headers at CDN/hosting layer:
  - `Strict-Transport-Security`
  - `X-Content-Type-Options: nosniff`
  - `X-Frame-Options: DENY`
  - `Referrer-Policy: strict-origin-when-cross-origin`
  - `Permissions-Policy` with least privilege
- Add/maintain CSP appropriate for Nuxt SPA and current external endpoints (AWS + MongoDB).
- Keep S3 bucket private for app data; only use scoped IAM permissions through Cognito identities.

### 9.11 Observability & Operational Checks
- Deployment health checks should include:
  - App bootstrap loads without console/runtime errors.
  - Cognito authentication flow works.
  - S3 save/load path works.
  - Bedrock challenge generation works (or expected local fallback is shown).
  - MongoDB settings read/write works (or expected localStorage fallback is used).
- Eventing and edge checks should include:
  - EventBridge rule received deployment-success event.
  - Lambda executed successfully with no permission/runtime failures.
  - CloudFront invalidation completed within expected time.
- Track platform metrics (build failures, CDN error rates, auth failures, API call errors) via AWS monitoring tools.
